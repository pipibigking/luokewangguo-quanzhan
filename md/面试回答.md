### 4. 精灵展示涉及多种筛选条件（属性、稀有度、价格等），你是如何实现筛选逻辑的？有没有做性能优化？

**回答：**

筛选逻辑实现方式：

1. **数据结构设计**：将筛选条件存储在一个 reactive 对象中
2. **计算属性过滤**：使用 computed 监听筛选条件变化，实时过滤数据
3. **防抖处理**：对输入类筛选（如价格范围）添加防抖，避免频繁计算

性能优化措施：

- 使用 computed 缓存计算结果，避免重复计算
- 对大量数据采用虚拟滚动（虽然当前数据量不大，但预留了扩展空间）
- 筛选条件变更时进行条件合并，减少 API 调用次数

核心代码逻辑：

```typescript
const filteredPets = computed(() => {
  return pets.value.filter(pet => {
    const matchAttr = !filter.attr || pet.attribute === filter.attr
    const matchRarity = !filter.rarity || pet.rarity === filter.rarity
    const matchPrice = pet.price >= filter.minPrice && pet.price <= filter.maxPrice
    return matchAttr && matchRarity && matchPrice
  })
})
```

### 15. JavaScript中闭包是什么？请举一个实际应用场景。

**回答：**

闭包是指函数能够访问其词法作用域之外的变量。

**特点：**

- 函数嵌套函数
- 内部函数可以访问外部函数的变量
- 参数和变量不会被垃圾回收

**实际应用场景**：

```javascript
// 防抖函数
function debounce(fn, delay) {
  let timer = null
  return function(...args) {
    clearTimeout(timer)
    timer = setTimeout(() => {
      fn.apply(this, args)
    }, delay)
  }
}

// 使用
const debouncedSearch = debounce(search, 300)
```

闭包在这里的作用是保存 timer 变量，使其在多次调用间保持状态。

### 16. Promise的三种状态是什么？如何处理Promise.all的错误？

**回答：**

Promise 的三种状态：

1. **Pending（进行中）**：初始状态
2. **Fulfilled（已成功）**：操作成功完成
3. **Rejected（已失败）**：操作失败

**Promise.all 错误处理**：

```javascript
Promise.all([promise1, promise2, promise3])
  .then(results => {
    console.log('全部成功:', results)
  })
  .catch(error => {
    console.log('至少一个失败:', error)
  })
```

如果需要全部结果（包括失败的），可以使用 `Promise.allSettled`：

```javascript
Promise.allSettled([promise1, promise2, promise3])
  .then(results => {
    results.forEach(result => {
      if (result.status === 'fulfilled') {
        console.log('成功:', result.value)
      } else {
        console.log('失败:', result.reason)
      }
    })
  })
```

### 17. 什么是事件循环（Event Loop）？宏任务和微任务的区别是什么？

**回答：**

**事件循环**是 JavaScript 处理异步操作的机制，包括调用栈、任务队列等。

**宏任务（Macro Task）**：

- setTimeout、setInterval
- I/O 操作
- UI 渲染
- script 整体代码

**微任务（Micro Task）**：

- Promise.then/catch/finally
- MutationObserver
- queueMicrotask

**执行顺序**：

1. 执行同步代码（宏任务）
2. 执行当前所有微任务
3. 执行下一个宏任务
4. 重复步骤 2-3

### 20. 什么是防抖（debounce）和节流（throttle）？分别用于什么场景？

**回答：**

**防抖（Debounce）**：

- 触发事件后等待一段时间再执行
- 如果期间再次触发，重新计时
- **场景**：搜索框输入、窗口 resize

```javascript
function debounce(fn, delay) {
  let timer = null
  return function(...args) {
    clearTimeout(timer)
    timer = setTimeout(() => fn.apply(this, args), delay)
  }
}
```

**节流（Throttle）**：

- 固定时间间隔内只执行一次
- **场景**：滚动事件、鼠标移动

```javascript
function throttle(fn, delay) {
  let lastTime = 0
  return function(...args) {
    const now = Date.now()
    if (now - lastTime >= delay) {
      lastTime = now
      fn.apply(this, args)
    }
  }
}
```

### 21. HTTP状态码304表示什么？它是如何实现的？

**回答：**

**304 Not Modified** 表示资源未被修改，客户端可以使用缓存版本。

**实现机制**：

1. 客户端第一次请求时，服务器返回资源并附带响应头：
   - `ETag`: 资源标识（如文件哈希）
   - `Last-Modified`: 最后修改时间
2. 客户端再次请求时，发送请求头：
   - `If-None-Match`: 上次收到的 ETag
   - `If-Modified-Since`: 上次收到的 Last-Modified
3. 服务器对比：
   - 如果资源未修改，返回 304
   - 如果资源已修改，返回 200 和新资源

**作用**：减少不必要的数据传输，提升性能。

### 22. TCP三次握手的过程是怎样的？为什么需要三次握手？

**回答：**

**三次握手过程**：

1. **第一次**：客户端发送 SYN（同步）包给服务器
2. **第二次**：服务器返回 SYN + ACK（确认）包
3. **第三次**：客户端发送 ACK 包给服务器

**为什么需要三次握手**：

1. **确认双方收发能力**：确保客户端和服务器都能发送和接收数据
2. **同步序列号**：双方协商初始序列号，用于可靠传输
3. **防止已失效的连接请求到达服务器**：避免服务器为无效连接分配资源

### 24. Composition API相比Options API有什么优势？

**回答：**

**Composition API 优势**：

1. **更好的代码组织**：按功能组织代码，而非按选项类型
2. **更好的逻辑复用**：通过组合函数复用逻辑，无需 mixins
3. **更好的类型推断**：TypeScript 支持更好
4. **更小的生产包**：Tree Shaking 更有效
5. **更灵活的代码组织**：不受选项约束

**示例对比**：

```javascript
// Options API
export default {
  data() { return { count: 0 } },
  methods: { increment() { this.count++ } },
  computed: { double() { return this.count * 2 } }
}

// Composition API
export default {
  setup() {
    const count = ref(0)
    const increment = () => count.value++
    const double = computed(() => count.value * 2)
    return { count, increment, double }
  }
}
```

### 25. ref和reactive的区别是什么？什么时候该用哪个？

**回答：**

**ref**：

- 用于基本类型（string、number、boolean）
- 需要通过 `.value` 访问和修改值
- 可以用于对象，但建议用 reactive

**reactive**：

- 用于对象和数组
- 直接访问属性，无需 `.value`
- 深层响应式

**使用建议**：

- 基本类型用 ref
- 对象类型用 reactive
- ref 可以持有任何类型，reactive 只能用于对象

```javascript
// ref
const count = ref(0)
count.value++

// reactive
const state = reactive({ count: 0 })
state.count++
```

### 26. 计算属性（computed）和侦听器（watch）的区别是什么？

**回答：**

**computed（计算属性）**：

- 用于推导衍生值
- 有缓存，依赖变化时才重新计算
- 同步执行
- **场景**：数据转换、过滤、组合

**watch（侦听器）**：

- 用于响应数据变化执行副作用
- 无缓存，每次变化都执行
- 支持异步
- **场景**：数据变化后的API调用、DOM操作

**示例**：

```javascript
// computed
const fullName = computed(() => `${firstName.value} ${lastName.value}`)

// watch
watch(count, (newVal, oldVal) => {
  console.log(`count changed from ${oldVal} to ${newVal}`)
})
```

### 27. Vue的虚拟DOM是如何工作的？diff算法的核心思想是什么？

**回答：**

**虚拟DOM工作流程**：

1. 将真实DOM转换为虚拟DOM（VNode树）
2. 状态变化时生成新的虚拟DOM
3. 通过diff算法比较新旧虚拟DOM
4. 只更新变化的部分到真实DOM

**diff算法核心思想**：

1. **同级比较**：只比较同层节点，不跨层比较
2. **key优化**：使用key标识节点，提高比较效率
3. **最小化操作**：通过移动、删除、创建实现高效更新

**优化策略**：

- 相同类型节点进行属性比较
- 不同类型节点直接替换
- 列表比较使用双端对比算法

## 四、算法与数据结构

### 33. 介绍一下快速排序的原理和时间复杂度。

**回答：**

**快速排序原理**：

1. 选择一个基准元素（pivot）
2. 将数组分为两部分：小于基准的元素和大于基准的元素
3. 递归排序左右两部分
4. 合并结果

**时间复杂度**：

- **平均情况**：O(n log n)
- **最坏情况**：O(n²)（已排序数组，每次选最小/最大元素）
- **最好情况**：O(n log n)

**空间复杂度**：O(log n)（递归调用栈）

**特点**：

- 原地排序，空间效率高
- 不稳定排序
- 实际应用中通常比归并排序快

### 34. 什么是二叉树的前序、中序、后序遍历？递归和迭代实现的区别？

**回答：**

**遍历方式**：

- **前序遍历**：根 → 左 → 右
- **中序遍历**：左 → 根 → 右（二叉搜索树中序遍历是有序的）
- **后序遍历**：左 → 右 → 根

**递归 vs 迭代**：

| 特性    | 递归         | 迭代   |
| ----- | ---------- | ---- |
| 代码复杂度 | 简单         | 复杂   |
| 空间复杂度 | O(h)（h为树高） | O(h) |
| 栈溢出风险 | 有（深度大时）    | 无    |
| 性能    | 略差（函数调用开销） | 略好   |

### 35. 哈希表的原理是什么？如何解决哈希冲突？

**回答：**

**哈希表原理**：

通过哈希函数将键映射到数组索引，实现 O(1) 时间复杂度的查找、插入、删除。

**哈希函数要求**：

- 相同输入必须产生相同输出
- 不同输入尽量产生不同输出
- 计算效率高

**解决哈希冲突的方法**：

1. **链地址法**：每个桶存储一个链表，冲突元素追加到链表
2. **开放寻址法**：冲突时探测下一个可用位置（线性探测、二次探测）
3. **再哈希法**：使用多个哈希函数

**负载因子**：元素数量 / 桶数量，通常保持在 0.7 以下。

### 36. 什么是栈和队列？它们的应用场景有哪些？

**回答：**

**栈（Stack）**：

- 后进先出（LIFO）
- 操作：push（入栈）、pop（出栈）、peek（查看栈顶）
- **场景**：函数调用栈、表达式求值、括号匹配、撤销操作

**队列（Queue）**：

- 先进先出（FIFO）
- 操作：enqueue（入队）、dequeue（出队）、front（查看队头）
- **场景**：任务队列、消息队列、BFS 遍历

**双端队列（Deque）**：两端都可以插入和删除，结合了栈和队列的特性。

### 37. 动态规划和贪心算法的区别是什么？请举例说明。

**回答：**

**动态规划（DP）**：

- 分解问题为子问题，保存子问题的解
- 自底向上或自顶向下求解
- **特点**：全局最优解，考虑所有可能

**贪心算法**：

- 每一步选择当前最优
- 不考虑后续影响
- **特点**：局部最优，不一定全局最优

**示例对比**：

| 问题      | 动态规划   | 贪心算法   |
| ------- | ------ | ------ |
| 背包问题    | 0-1背包  | 分数背包   |
| 找零钱     | 复杂金额体系 | 标准金额体系 |
| 最长公共子序列 | DP     | 不适用    |

### 38. 如何判断链表是否有环？请写出算法思路。

**回答：**

**方法一：哈希表**

遍历链表，将每个节点存入哈希表，如果遇到重复节点则有环。

- 时间复杂度：O(n)
- 空间复杂度：O(n)

**方法二：快慢指针（Floyd判圈算法）**

使用两个指针，快指针每次走两步，慢指针每次走一步。如果相遇则有环。

- 时间复杂度：O(n)
- 空间复杂度：O(1)

**算法思路**：

```javascript
function hasCycle(head) {
  if (!head || !head.next) return false
  
  let slow = head
  let fast = head.next
  
  while (fast && fast.next) {
    if (slow === fast) return true
    slow = slow.next
    fast = fast.next.next
  }
  
  return false
}
```

### 39. 二分查找的前提条件是什么？时间复杂度是多少？

**回答：**

**前提条件**：

1. 数组必须有序（升序或降序）
2. 数组支持随机访问（如数组，不是链表）

**时间复杂度**：

- **平均/最坏情况**：O(log n)
- **最好情况**：O(1)（目标在中间）

**空间复杂度**：

- 递归实现：O(log n)
- 迭代实现：O(1)

**实现思路**：

```javascript
function binarySearch(arr, target) {
  let left = 0
  let right = arr.length - 1
  
  while (left <= right) {
    const mid = Math.floor((left + right) / 2)
    if (arr[mid] === target) return mid
    if (arr[mid] < target) left = mid + 1
    else right = mid - 1
  }
  
  return -1
}
```

### 40. 什么是图的深度优先遍历和广度优先遍历？

**回答：**

**深度优先遍历（DFS）**：

- 从起点出发，尽可能深地访问节点
- 遇到死路时回溯
- **实现方式**：递归或栈
- **应用**：拓扑排序、强连通分量

```javascript
// 递归实现
function dfs(node, visited) {
  visited.add(node)
  for (const neighbor of node.neighbors) {
    if (!visited.has(neighbor)) {
      dfs(neighbor, visited)
    }
  }
}
```

**广度优先遍历（BFS）**：

- 从起点出发，逐层访问节点
- 使用队列存储待访问节点
- **应用**：最短路径（无权图）

```javascript
function bfs(start) {
  const queue = [start]
  const visited = new Set([start])
  
  while (queue.length) {
    const node = queue.shift()
    for (const neighbor of node.neighbors) {
      if (!visited.has(neighbor)) {
        visited.add(neighbor)
        queue.push(neighbor)
      }
    }
  }
}
```
