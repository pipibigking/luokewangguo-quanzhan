# 后台登录私人化改造方案

## 🔍 问题分析

当前登录机制的问题：

```
登录成功 → localStorage.setItem("admin_authenticated", "true")
路由守卫 → localStorage.getItem("admin_authenticated") === "true"
```

**核心缺陷**：登录凭证是一个全局布尔值 `"true"`，没有任何用户特征、没有过期时间、无法区分不同用户。同一台电脑上任何人打开网站都能直接进入后台。

## 🎯 目标

* **已登录用户**：浏览器记住登录状态，直接进入后台（同当前体验）

* **未登录用户**：必须经过账号密码验证才能进入后台

* **私人化**：每个浏览器/设备有独立的登录凭证，互不影响

## 🛠️ 方案：JWT Token 认证

引入 JWT（JSON Web Token）替代简单的布尔标记：

```
登录流程（改造后）：
用户输入账号密码 → 后端验证 → 生成 JWT Token（含用户名 + 过期时间） → 返回前端
前端存 Token → 路由守卫检查 Token → 进入后台
请求 API 时自动带 Token → 后端验证 Token → 返回数据
```

### 为什么选 JWT

* ✅ 无状态，不需要数据库存储会话

* ✅ 可设过期时间（7天），实现"记住登录"

* ✅ 包含用户信息，可区分不同管理员

* ✅ 标准安全机制，防止伪造

***

## 📋 实施步骤

### Step 1：后端 — 安装依赖

在 `backend/requirements.txt` 添加：

```
python-jose[cryptography]==3.3.0
```

> `python-jose` 用于 JWT 的生成和验证，`cryptography` 提供加密支持。

### Step 2：后端 — 添加 JWT 配置和工具函数

在 `backend/main.py` 顶部附近添加：

* `SECRET_KEY`（随机密钥，用于签名 Token）

* `ALGORITHM = "HS256"`

* `ACCESS_TOKEN_EXPIRE_DAYS = 7`（Token 7 天过期）

* `create_access_token(data)` — 生成 Token

* `verify_access_token(token)` — 验证 Token，返回 payload 或 None

### Step 3：后端 — 修改登录接口

**`POST /api/admin/login`** 改造：

* 登录成功后，调用 `create_access_token({"sub": username})` 生成 Token

* 返回数据增加 `token` 字段：

```json
{
  "success": true,
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "username": "admin",
  "message": "登录成功"
}
```

### Step 4：后端 — 添加 Token 验证接口

**`GET /api/admin/verify`**（新增）：

* 前端路由守卫可调用此接口验证 Token 有效性

* 从 `Authorization` 请求头中取 Token

* 返回 `{ "valid": true, "username": "xxx" }` 或 401

### Step 5：后端 — 添加 API 认证中间件

用 FastAPI middleware 统一保护管理端 API：

1. 创建 `get_token_from_header(request)` 辅助函数
2. 创建 middleware，对需要保护的路径进行 Token 校验
3. **保护的路径规则**：

   * 所有 `/api/admin/*` 路径（排除 `/api/admin/login`）

   * 所有写入操作（POST/PUT/PATCH/DELETE）的 `/api/*` 路径

### Step 6：前端 — 修改认证工具 (`auth.ts`)

| 函数                  | 改造内容                                         |
| ------------------- | -------------------------------------------- |
| `AUTH_KEY`          | 从 `"admin_authenticated"` 改为 `"admin_token"` |
| `login()`           | 成功后存 Token 到 localStorage，而非 `"true"`        |
| `isAuthenticated()` | 检查 Token 是否存在且非空，而非检查 `=== 'true'`           |
| `logout()`          | 清除 Token                                     |
| 新增 `getToken()`     | 返回 localStorage 中的 Token                     |
| 新增 `verifyToken()`  | 调用后端 `/api/admin/verify` 验证 Token            |

### Step 7：前端 — 添加请求拦截器 (`api/index.ts`)

给 axios 实例添加请求拦截器：

* 从 localStorage 读取 Token

* 在请求头自动添加 `Authorization: Bearer <token>`

### Step 8：前端 — 完善路由守卫 (`router/index.ts`)

* 当前：只检查 `isAuthenticated()` 返回布尔值

* 改造后：先检查本地 Token 存在，再可选调用后端验证

### Step 9：前端 — 登录页面适配 (`LoginPage.vue`)

* `handleLogin()` 调用 `login()` 成功后跳转 `/admin`

* 基本逻辑不变，底层已由 `auth.ts` 封装

### Step 10：测试验证

1. ✅ 首次登录：输入账号密码 → 登录成功 → 进入后台
2. ✅ 关闭浏览器重新打开 → 直接进入后台（Token 未过期）
3. ✅ 登出 → 清除 Token → 重新进入需登录
4. ✅ 不同浏览器/设备 → 互不影响，各自需登录
5. ✅ Token 过期（7天后）→ 需重新登录

## 📁 涉及修改的文件清单

| 文件                             | 改动类型                                    |
| ------------------------------ | --------------------------------------- |
| `backend/requirements.txt`     | 新增依赖                                    |
| `backend/main.py`              | 新增 JWT 工具函数、修改登录接口、新增 verify 接口、添加认证中间件 |
| `frontend/src/utils/auth.ts`   | 重写认证逻辑                                  |
| `frontend/src/api/index.ts`    | 添加请求拦截器                                 |
| `frontend/src/router/index.ts` | 路由守卫增强（可选）                              |

## ⚠️ 注意事项

1. **向后兼容**：原有 `admin_authenticated` 标记会被清除，所有用户首次需要重新登录一次
2. **Token 安全**：Token 存 localStorage，建议后续生产环境启用 HTTPS
3. **密码存储**：当前为明文，本次不改动密码存储方式，聚焦登录会话管理
